# RDRAで要件定義を行うためのフェーズ
0_ZeroOne指示.md内でphase毎の出力フォルダーが指示され、phase毎に出力するフォルダーは分かれる

## **RDRAZeroOneは段階的に要件を定義するために４つのフェーズですすめる**
各フェーズでモデルを洗い出すときは「RDRA構造の説明」を十分に理解し洗い出す
実行はフェーズ毎に行い、複数のフェーズを同時に実行しない
【重要】の指示のあるものは必ず実行する

## ファイル出力ルール
- 同じファイルが既に存在する場合は、ファイル全体を上書きする
- 出力するファイルには必ずヘッダー（カラム名）行を出力する
  
/*
    フェーズ１ -----　業務と管理する情報・状態を洗い出す　----------
*/
## フェーズ１
### フェーズ１の目的
初期要望からシステム化対象を３つの視点で分析する
- ビジネスを関わる登場人物と業務
- ビジネスの関心事をポリシーとパラメータ
- システムを情報と状態モデルで表現
この３つの視点をシステム化の出発点とする

### 目的を達成するために次のことを行う
- まず初めに「初期要望.txt」を読込、十分に理解してください
- 「初期要望.txt」のシステム要求から必要な「業務」を洗い出してください
- 「業務」に関わる「アクター」（システムを利用する人の役割）を見つけ、「アクター.tsv」ファイルのフォーマットで出力する
- システムやサービス(SaaS)は「アクター」に含めないでください
- 「業務」に関わる「外部システム」（システムに関わる外部のシステムの役割）を見つけ、「外部システム.tsv」ファイルのフォーマットで出力する
- 「業務」が従うべき「ビジネスポリシー」を洗い出し、「ビジネスポリシー.tsv」ファイルのフォーマットで出力する
- 「業務」の拡張性・柔軟性を生み出す「ビジネスパラメータ」を洗い出し、「ビジネスパラメータ.tsv」ファイルのフォーマットで出力する
- ビジネスポリシーを元に「業務」を見直す
- 洗い出した「業務」と「ビジネスポリシー」からを参考に「業務」を行うアクターと関係する外部システムを検討し、必要な「BUC」を洗出す
- 洗い出した「BUC」に関わる価値を得るアクター毎に「BUC」を分ける
- 業務において想定される例外的な事態に対応するための「BUC」も洗い出す
- 洗い出した業務とBUCは「業務.tsv」ファイルのフォーマットで出力する
- 「BUC」を遂行するために必要な「情報」を洗い出し、「情報.tsv」ファイルのフォーマットで出力する
- 「BUC」を管理するために必要な「状態モデル」を洗い出し、「状態.tsv」ファイルのフォーマットで出力する

### 「アクター.tsv」ファイルのフォーマット
システムに関わる人の役割を定義する
「アクター⇒役割」　"⇒":タブ区切りに変える
- アクター：システムに関わる人の役割名
- アクター名は「管理者」や「担当者」などの単純な名前ではなく、役割が分かる名前にする
- 役割：アクターの役割の内容
### 「外部システム.tsv」ファイルのフォーマット
システムに関わる外部のシステムを定義する
「外部システム⇒役割」　"⇒":タブ区切りに変える
- 外部システム：システムに関わる外部のシステム
- 外部システム名は役割が分かる名前にする
- 役割：外部システムの役割の内容説明
### 「ビジネスポリシー.tsv」ファイルのフォーマット
自社独自ビジネスに対するポリシーを定義する
「ビジネスポリシー⇒説明」　"⇒":タブ区切りに変える
- ビジネスポリシー：システムに関わる外部のシステム
- 説明：ビジネスポリシーの説明
### 「ビジネスパラメータ.tsv」ファイルのフォーマット
ビジネスに柔軟性を持たせるために区分や種別を表すビジネスパラメータを定義する
「ビジネスパラメータ⇒説明」　"⇒":タブ区切りに変える
- ビジネスパラメータ：ビジネスパラメータの名前
- 説明：ビジネスパラメータの説明と値
### 「業務.tsv」ファイルのフォーマット
システム化対象の業務を定義する
「業務⇒BUC⇒説明」　"⇒":タブ区切りに変える
- 業務：業務を分類する名前　例：「販売管理」「在庫管理」など
- BUC：洗い出した「BUC」の名前
- 説明：BUCの内容を説明
### 「情報.tsv」ファイルのフォーマット
BUCを進めるうえで必要な情報を元にシステムで扱う情報を表す
「情報⇒説明」　"⇒":タブ区切りに変える
- 情報：ユニークなIDで一意に識別できるデータを表す
- 説明：情報が必要な理由を説明
###「状態.tsv」ファイルのフォーマット
ビジネスを駆動するために管理する必要のある状態を表す
「状態モデル⇒説明」　"⇒":タブ区切りに変える
- 状態モデル：状態遷移の管理単位となる
- 説明：状態モデルが必要な理由と管理するものを説明

    ビジネスパラメータ.tsvを再作成する
}
if フェーズ１で作成したファイル群 not exist 業務.tsv {
    業務.tsvを再作成する
}
if フェーズ１で作成したファイル群 not exist 情報.tsv {
    情報.tsvを再作成する
}

## フェーズ２
### フェーズ２の目的
フェーズ１の出力を出発点にビジネスの詳細化し、ビジネスの関心事はビジネスルールとして整理、システムの情報と状態モデルは構造化する

### 目的を達成するためフェーズ１の結果を元に以下の指示に従ってください
#### 「業務」の作成
- フェーズ１の「業務.tsv」を元に「BUC」の構成要素となる「アクティビティ」（仕事）を追記する
- 「BUC」の「アクティビティ」は例外的な事柄も考慮し、その対応も追記する
- 「アクティビティ」（仕事）にユースケース（UC）を結びつける
- ユースケースの名前は行う対象（XX）と行うこと（YY）を一つの名前として組立てる　「XXをYYする」形式
- 洗い出した「アクティビティ」、「ユースケース」を、「業務.tsv」ファイルのフォーマットで出力する
#### 「情報」の作成
- フェーズ１の「情報.tsv」の各「情報」について属性を洗出す
- 情報は相互に関係するので、各「情報」毎に関係する情報を関連情報として出力する
- 洗い出した属性と関連情報を「情報.tsv」ファイルのフォーマットで出力する
#### 「条件」の作成
- システムが従うべきルールを「条件」として洗い出し、「条件.tsv」ファイルのフォーマットで出力する
#### 「条件」の作成
- フェーズ１の「ビジネスポリシー.tsv」を参考にビジネスルールを洗い出し「条件.tsv」ファイルのフォーマットで出力する
#### 「条件」の作成
- フェーズ１の「ビジネスパラメータ.tsv」を参考にシステムで管理すべき区分や種別を「バリエーション」として洗い出し「バリエーション.tsv」ファイルのフォーマットで出力する
#### 「状態」の作成
- フェーズ１の「状態.tsv」の各「状態モデル」について管理する状態を洗出し、状態遷移を作り上げる
- 各「状態モデル」の状態遷移に洗い出したユースケース（UC）を割り当てる
- 洗い出した状態モデル毎に「状態.tsv」ファイルのフォーマットで出力する

### 「業務.tsv」ファイルのフォーマット
洗い出した業務とBUCは以下のフォーマットで出力する
「業務⇒BUC⇒アクティビティ⇒ユースケース⇒説明」　"⇒":タブ区切りに変える
- 業務：業務を分類する名前　例：「販売管理」「在庫管理」など
- BUC：洗い出したBUCの名前
- アクティビティ：洗い出したアクティビティの名前　アクティビティは仕事を表す
- ユースケース（UC）：洗い出したユースケースの名前　名前は「（主語）を（動詞）する」形式にする
- 説明：BUCの内容を説明

### 「情報.tsv」ファイルのフォーマット
洗い出した情報は以下のフォーマットで出力する
「情報⇒属性⇒関連情報⇒活用法」　"⇒":タブ区切りに変える
- 情報：ユニークなIDで一意に識別できるデータを表す
- 属性：情報が扱う属性を表し、「、」区切りで複数の属性を一つの文字列で表す
- 関連情報：情報間の関係を表し、「、」区切りで複数の情報を一つの文字列で表す
- 活用法：情報の活用方法を説明
### 「状態.tsv」ファイルのフォーマット
洗い出した状態モデルは以下のフォーマットで遷移毎に１行で出力する
「状態モデル⇒遷移前状態⇒遷移後状態⇒説明」　"⇒":タブ区切りに変える
- 状態モデル：状態遷移の管理単位で情報に対応する
- 遷移前状態：状態モデルが管理する遷移前の状態
- 遷移後状態：状態モデルが管理する遷移後の状態
- 説明：遷移された状態を表す
### 「条件.tsv」ファイルのフォーマット
洗い出した条件は以下のフォーマットで出力する
「条件⇒説明」　"⇒":タブ区切りに変える
- 条件：ビジネスポリシーを実現するためにユースケースで従うべきルールを記述する
- 説明：仕事の内容を掘り下げるための要求を表す
### 「バリエーション.tsv」ファイルのフォーマット
洗い出したバリエーションは以下のフォーマットで出力する
「バリエーション⇒値⇒説明」　"⇒":タブ区切りに変える
- バリエーション：区分や種別を表すバリエーション
- 値：バリエーションの値の集合を表す、「、」区切りで複数の値を一つの文字列で表す
- 説明：バリエーションの役割と活用用途を表す

### 【重要】実行結果の確認
if フェーズ２で作成したファイル群 not exist 業務.tsv {
    業務.tsvを再作成する
}
if フェーズ２で作成したファイル群　not exist 条件.tsv {
    条件.tsvを再作成する
}
if フェーズ２で作成したファイル群　not exist バリエーション.tsv {
    バリエーション.tsvを再作成する
}
if フェーズ２で作成したファイル群　not exist 情報.tsv {
    情報.tsvを再作成する
}
if フェーズ２で作成したファイル群　not exist 状態.tsv {
    状態.tsvを再作成する
}

## フェーズ３
### フェーズ３の目的
フェーズ２の出力を元にユースケースに情報や条件、関わるアクターや外部システムをつなげることでユースケースが行うことを表す

### 目的を達成するためフェーズ２の結果を元に以下のStepを実行してください

#### Step1:フェーズ２の「情報.tsv」「状態.tsv」「条件.tsv」「バリエーション.tsv」を元に以下の指示に従ってください
- 同じ目的の「情報」をグループ化したものを「コンテキスト」とし、「情報.tsv」に従って出力する
- 「状態モデル」に関わる「情報」の「コンテキスト」を「状態モデル」の「コンテキスト」とし、「状態.tsv」ファイルのフォーマットに従って出力する
- 「条件」の縦軸/横軸に「情報」が関わる場合に「情報」の「コンテキスト」を「条件」を「コンテキスト」とし、「条件.tsv」ファイルのフォーマットに従って出力する
- 「条件」に関係する「情報」がない場合は「条件」をグルーピング化してその条件の「コンテキスト」にし、「条件.tsv」ファイルのフォーマットに従って出力する
- 「バリエーション」が「条件」の縦軸/横軸に関わる場合に「条件」の「コンテキスト」を「バリエーション」の「コンテキスト」とする
- 「バリエーション」に関係する「条件」がない場合は「バリエーション」をグルーピング化し、それを「コンテキスト」にする
- 「バリエーション.tsv」ファイルのフォーマットに従って出力する

##### 「情報.tsv」ファイルのフォーマット
洗い出した情報は以下のフォーマットで出力する
「コンテキスト⇒情報⇒属性⇒関連情報⇒説明」　"⇒":タブ区切りに変える
- コンテキスト：情報をまとめる単位
- 情報：ユニークなIDで一意に識別できる
- 属性：情報が扱う属性を表します、「、」区切りで複数の属性を一つの文字列で表す
- 関連情報：情報間の関係を表します、「、」区切りで複数の情報を一つの文字列で表す
- 説明：情報の説明

##### 「状態.tsv」ファイルのフォーマット
洗い出した状態モデルは以下のフォーマットで出力する
「コンテキスト⇒状態モデル⇒遷移前状態⇒遷移後状態⇒説明」　"⇒":タブ区切りに変える
- コンテキスト：状態モデルをまとめる単位
- 状態モデル：状態遷移の管理単位で情報に対応する
- 遷移前状態：状態モデルが管理する遷移前の状態
- 遷移後状態：状態モデルが管理する遷移後の状態
- 説明：遷移された状態を表す

##### 「条件.tsv」ファイルのフォーマット
洗い出した条件は以下のフォーマットで出力する
「コンテキスト⇒条件⇒説明」　"⇒":タブ区切りに変える
- コンテキスト：条件が所属するコンテキストを表す
- 条件：ビジネスポリシーを実現するためにユースケースで従うべきルールを記述する
- 説明：仕事の内容を掘り下げるための要求を表す

##### 「バリエーション.tsv」ファイルのフォーマット
洗い出したバリエーションは以下のフォーマットで出力する
「コンテキスト⇒バリエーション⇒値⇒説明」　"⇒":タブ区切りに変える
- コンテキスト：バリエーションが所属するコンテキストを表す
- バリエーション：区分や種別などの業務上の分類を表し、条件で使われる
- 値：区分や種別を表す、「、」区切りで複数の値を一つの文字列で表す
- 説明：バリエーションを表す

#### Step2:アクターのグループの作成
フェーズ１の「アクター.tsv」を入力データとして「アクター」ファイルのフォーマットで出力してください
「アクター」は、業務の実施に関わる人や組織を指します。
アクターをまとめたグループを作成して「アクター群」としてください
アクターと説明は入力データをそのまま出力してください

##### 「アクター.tsv」ファイルのフォーマット
システムに関わる人の役割を定義する
「アクター群⇒アクター⇒説明」　"⇒":タブ区切りに変える
- アクター群：アクターをまとめるグループの名前　
- アクター：洗い出されたアクター名
- 説明：アクターの説明

#### Step3:外部システムのグループ化
フェーズ１の「外部システム.tsv」を入力データとして「外部システム」ファイルのフォーマットで出力してください
「外部システム」は、業務の実施に関わる外部のシステムを指します。
外部システムをまとめたグループを作成して「外部システム群」としてください
外部システムと説明は入力データをそのまま出力してください

##### 「外部システム.tsv」ファイルのフォーマット
システムに関わる外部のシステムを定義する
「外部システム群⇒外部システム⇒説明」　"⇒":タブ区切りに変える
- 外部システム群：外部システムをまとめるグループの名前
- 外部システム：システムに関係する外部のシステム名
- 説明：外部システムの説明

#### Step4:フェーズ２の「業務.tsv」を元に「BUC.tsv」ファイルを作成する
以下の指示に従って「BUC.tsv」ファイルを出力する
- 「業務.tsv」内の行ごとのユースケース（UC）の関係を構築する
- ユースケースが操作する「情報」を関連付ける。「情報」は「情報.tsv」に出力したものから設定する
- ユースケースが従う「条件」を関連付ける。「条件」は「条件.tsv」に出力したものから設定する
- ユースケースに関わるアクターを洗い出す。「アクター.tsv」に出力したものから設定する
- ユースケースに関わる外部システムを洗い出す。「外部システム.tsv」に出力したものから設定する

##### 「BUC.tsv」ファイルのフォーマット
洗い出したBUCは以下のフォーマットで出力する
「業務⇒BUC⇒アクティビティ⇒ユースケース⇒情報⇒条件⇒アクター⇒外部システム⇒説明」　"⇒":タブ区切りに変える
- 業務：「業務.tsv」の業務をセットする
- BUC：業務フローの単位となり、業務内に複数存在する
- アクティビティ：業務フロー内の各アクティビティを表す
- ユースケース（UC）：上記アクティビティに対応し、システムが行うことをユースケースとして表す
- 情報：ユースケースにつながる情報を表す、「、」区切りで複数の情報を一つの文字列で表す
- 条件：ユースケースにつながる条件を表す、「、」区切りで複数の条件を一つの文字列で表す
- アクター：ユースケースにつながるアクターを表す、「、」区切りで複数のアクターを一つの文字列で表す
- 外部システム：ユースケースにつながる外部システムを表す、「、」区切りで複数の外部システムを一つの文字列で表す
- 説明：「ユースケース（UC）」の説明

if フェーズ３で作成したファイル群 not exist BUC.tsv {
    BUC.tsvを再作成する
}

## フェーズ４：
### フェーズ４の目的
フェーズ３の出力を元にユースケースに関わる要素をより柔軟な構造の関連モデルと関連オブジェクトの形式に変換し、
情報や状態モデルに関係するものをつなげる

### 目的を達成するためフェーズ３の結果を元に以下のStepを実行してください

#### Step1:「情報」ファイルの関係作成
フェーズ３の「情報」を元に以下の処理を行う
情報に関係する状態モデルを関連付ける
情報はユニークなIDを持っている。状態モデルもユニークなIDの状態遷移を表す
同じIDをもつものを関連付ける
情報に従属するバリエーションを関連付ける
例：会員ランク（バリエーション）は会員（情報）に従属する
上記の操作を行い「情報」ファイルのフォーマットで出力する

##### 「情報.tsv」ファイルのフォーマット
関連付けした情報は以下のフォーマットで出力する
- コンテキスト：情報をまとめる単位
- 情報：ユニークなIDで一意に識別できる
- 属性：情報が扱う属性を表します、「、」区切りで複数の属性を一つの文字列で表す
- 関連情報：情報間の関係を表します、「、」区切りで複数の情報を一つの文字列で表す
- 状態モデル：状態で管理する状態モデルを表します、「、」区切りで複数の状態モデルを一つの文字列で表す
- バリエーション：情報に従属するバリエーションを表します、「、」区切りで複数のバリエーションを一つの文字列で表す
- 説明：情報の説明

#### Step2:「状態」ファイルの関係作成
フェーズ３の「状態モデル」を元に以下の処理を行う
- 状態には３種類の状態（最初の状態、途中の状態、最後の状態）があり、それぞれに出力方法が異なる
- 最初の状態：状態がないところから始めの状態を指す
- 途中の状態：状態から次の状態に遷移する
- 最後の状態：その状態から他の状態に遷移しない状態のことを指す

- 状態を遷移させるユースケースを「遷移UC」に割り当てる
- 最初の状態：「状態：空欄、遷移UC：空欄、遷移先状態：最初の状態」
- 途中の状態：「状態：遷移前の状態、遷移UC：該当UC名、遷移先状態：遷移後の状態」
- 最後の状態：「状態：最後の状態、遷移UC：空欄、遷移先状態：空欄」
上記の操作を行い「状態」ファイルのフォーマットで出力する

##### 「状態.tsv」ファイルのフォーマット
関連付けした状態モデルは以下のフォーマットで出力する
- コンテキスト：状態モデルが対応する情報と同じコンテキストを表す
- 状態モデル：状態遷移の管理単位で情報に対応する
- 状態：情報が取る状態を表す
- 遷移UC：状態を遷移させるユースケースを表す
- 遷移先状態：遷移された状態を表す
- 説明：状態の説明

#### Step3:「条件」ファイルの関係作成
マトリックス形式の条件の縦軸か横軸に当てはまる以下のものを当てはめる
・状態モデルが当てはめる場合は、状態モデルに対応する名前を設定する
・バリエーションが当てはまる場合は、バリエーションを当てはめる

##### 「条件.tsv」ファイルのフォーマット
関連付けした条件は以下のフォーマットで出力する
- コンテキスト：条件が所属するコンテキストを表す　条件に関わる情報のコンテキストを使う
- 条件：ビジネスルールの名前を表す
- 条件の説明：バリエーションを表す
- バリエーション：条件の縦軸/横軸となるバリエーションを表す、「、」区切りで複数のバリエーションを一つの文字列で表す
- 状態モデル：条件の縦軸/横軸となる状態モデルを表す、「、」区切りで複数の状態モデルを一つの文字列で表す

#### Step4:「BUC」ファイルのUCと関連するオブジェクトの構造変換
フェーズ３の「BUC.tsv」からフェーズ４のBUC形式（「BUC」ファイルのフォーマット）のファイルフォーマットに変換する
フェーズ３の「BUC.tsv」ファイルの１行の定義を「情報」「条件」「アクター」「外部システム」の各カラムに対応してフェーズ４のBUCでは行方向に変換します

##### フェーズ４の「BUC.tsv」ファイルのフォーマット
洗い出したBUCは以下のフォーマットで出力する
「業務⇒BUC⇒先⇒アクティビティ⇒次⇒ユースケース⇒関連モデル1⇒関連オブジェクト1⇒関連モデル2⇒関連オブジェクト2⇒説明」　"⇒":タブ区切りに変える
- 業務：「BUC.tsv」の業務をセットする
- BUC：「BUC.tsv」のBUCをセットする
- 先：""をセットする
- アクティビティ：業務フロー内の各アクティビティを表す
- 次：""をセットする
- ユースケース（UC）：「BUC.tsv」のユースケースをセットする
- 関連モデル1：次の候補[情報,条件,画面,イベント,タイマー]　この中の一つに決まる
- 関連オブジェクト1：関連モデル1のモデルに対応するオブジェクト
- 関連モデル2：次の候補[アクター,外部システム,""]　この中の一つに決まる
- 関連オブジェクト2：関連モデル2のモデル[[アクター,外部システム]に対応するオブジェクト、関連モデル2が""の時は""をセット
- 説明：「ユースケース（UC）」の説明

##### フェーズ３のBUCからフェーズ４のBUCフォーマットへの変換ルール
###### 変換ルール：
- 「業務」,「BUC」,「アクティビティ」,「ユースケース」,「説明」カラムの値はそのままコピーする
- 「先」「次」カラムには""をセットする
- それ以外のカラムは以下のルールに従う
- カラム内の値に「、」か「,」がある場合は値を「、」か「,」で分割し、分割した数に応じて複数行で出力する
- 「情報」カラムは「関連モデル1」を「情報」にし、「関連オブジェクト1」に「情報」カラムの値をセットし、「関連モデル2」と「関連オブジェクト2」にはブランクをセットする
- 「条件」カラムは「関連モデル1」を条件にし、「関連オブジェクト1」に「条件」カラムの値をセットし、「関連モデル2」と「関連オブジェクト2」にはブランクをセットする
- 「アクターカラム」は「関連モデル1」を「画面」にし、「関連オブジェクト1」に「画面名」をセット、「関連モデル2」を「アクター」にし、「関連オブジェクト2」に「アクター」カラムの値をセットする
- 「外部システム」カラムは「関連モデル1」を「イベント」にし、「関連オブジェクト1」に「イベント名」をセット、「関連モデル2」を「外部システム」にし、「関連オブジェクト2」に「外部システム」カラムの値をセットする
- 「画面名」は「UC」が提供する画面として適切な名前にする
- 「イベント名」は「UC」と「外部システム」との連携として適切な名前にする

###### 変換例：
####### フェーズ３のBUCファイル形式：

####### フェーズ４のBUC形式：
- 業務A/BUC_A/""/アクティビティ_A/""/UC_A/情報/情報A/""/""/説明A
- 業務A/BUC_A/""/アクティビティ_A/""/UC_A/情報/情報B/""/""/""
- 業務A/BUC_A/""/アクティビティ_A/""/UC_A/条件/条件A/""/""/""
- 業務A/BUC_A/""/アクティビティ_A/""/UC_A/条件/条件B/""/""/""
- 業務A/BUC_A/""/アクティビティ_A/""/UC_A/画面/画面A/アクター/アクターA/""
- 業務A/BUC_A/""/アクティビティ_A/""/UC_A/イベント/イベントA/外部システム/外部システムA/""

####### 変換例：フェーズ３の「情報、条件、アクター、外部システム」カラムからフェーズ４の「関連モデル1、関連オブジェクト1、関連モデル2、関連オブジェクト2」への変換
- フェーズ３ -> フェーズ４
